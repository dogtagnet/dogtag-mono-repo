## Mission

Build the **DogTag pet-credentialing ecosystem** exactly as specified in:
- **`docs/architecture.md`** — system + smart-contract architecture (the *what* and *why*).
- **`docs/implementation.md`** — per-function pseudocode, contract bodies, endpoints, Docker, deploy (the *how*).
- **`docs/research/`** — the evidence behind every decision (research briefs `01`–`13`, audits `01`–`12`, and `CHANGESPEC-v2`/`-v3`/`-v4`). **Normative precedence:** `architecture.md §13` and `implementation.md §11` override earlier sections; within §13/§11, the **highest-numbered sub-section wins** (the **v4.1** remediations **§13.9 / §11.10** are the latest — Poseidon unification + `rootIssuer[R]` clone resolution).

**These three are the source of truth. Read them in full before writing any code.** Where `architecture.md §13` and `implementation.md §11` (the audit remediations) conflict with anything earlier in those docs, **the remediation sections win** — they are normative.

The system: pet owners hold pets' verifiable credentials in native mobile apps; vets/groomers run **self-hosted** Docker stacks that issue credentials by anchoring salted-Merkle roots on the **ROAX** EVM chain (chainId 135, gas token PLASMA); a **central/admin** stack (we host) runs discovery, issuer whitelisting, and is the appointment system-of-record. Verification = **three authenticity pillars** (integrity + on-chain status + DNS identity) that gate validity for everyone, **plus a contextual `ownership` fragment** (`ownerOf(dogTagId)==userWalletAddress`) that gates *only the owner's own self-import* and is `NOT_APPLICABLE` for third-party verifiers.

**v3 adds a 4th capability — on-chain proof-of-verification.** Verifiers (groomer/vet/airline/gov) record a **consent-bound, on-chain attestation** that a credential was presented — keyed by `purpose` (`GROOMING_INTAKE`/`TRAVEL_PRESENTATION`/`AIRLINE_CHECKIN`/`VET_INTAKE`). Two paths: a **normal path** (EIP-712 `VerificationConsent` over the root `R`, ECDSA) and a **Groth16 ZK path** (consent over the same `R`, EdDSA-BabyJubjub, no raw data on chain). The mobile-user **self-import path is UNCHANGED** (`verify(..., mode:"self-import")`); verification and import are **decoupled**.

**v4 unifies the credential commitment on Poseidon.** The credential commitment — leaf hash, Merkle tree, and verification nullifier — is now a **SINGLE Poseidon root `R`** (one pinned circomlib BN254 instantiation), anchored at issuance via `issue(R)` and proven **directly** by the Groth16 circuit. There is **no `rKec`/`rZk` duality** anymore. **keccak is retained ONLY where the EVM/ECDSA standards mandate it:** EIP-712/ECDSA signature digests (`VerificationConsent`, `DogTagSBT.recover`), Ethereum address derivation, and pure namespacing keys (`recordType=keccak256(label)`, `VERIFY:` whitelist keys, clone `salt`). Everything that enters the credential commitment or the circuit is Poseidon.

## Non-negotiable principles

1. **Determinism first.** The credential commitment must produce **bit-identical Poseidon roots across FOUR environments — circom, TS, Rust, and Solidity** — on **ONE pinned circomlib BN254 Poseidon** (x⁵ S-box, `R_F=8`, per-`t` `R_P`, seed `"poseidon"`, circomlib MDS/round-constants). Pin the libs: circom → **circomlib** `Poseidon`; TS → **`poseidon-lite`**; Rust → **`light-poseidon`** (`new_circom(n)`, Veridise-audited); Solidity → **`poseidon-solidity`** (`PoseidonT3..T7`). **circomlib is the reference-of-record:** the per-arity anchor vectors are **generated from circomlib** (the circom `Poseidon` template / `circomlibjs`), and `poseidon-lite`/`light-poseidon`/`poseidon-solidity` are conformance-tested *against circomlib's outputs* — if any disagrees, **circomlib wins** and that lib is repinned or replaced. **CI MUST assert pinned anchor vectors at EACH arity used — t=2 (bytesToField fold), t=3 (Merkle node), t=6 (leaf), t=7 (nullifier) — bit-identical in all four** *before anything depends on the SDKs* (a lib can match at t=3 yet diverge at t=6/t=7 since `R_P`/constants are per-`t`; `poseidon([1,2]) = 0x115cc0f5...189a` is only the t=3 anchor; circomlibjs has historically drifted — pin + test). Pin ALL field reductions to the BN254 **scalar field `r`** (not base `q`). Build `crates/dogtag-standard-rs` and `packages/dogtag-standard-ts` against a **shared `testvectors.json`** and assert parity in CI. `encodeValue` is **unchanged** (pinned decimal/integer grammar, typed string input, NFC version pin, no-float guard, flatten/keyPath grammar, first-two-colons parse per `implementation.md §11.2`) — **only the final hash changes to Poseidon**, via `fieldOf` byte→field packing.
2. **Security gates are blocking.** Do **not** deploy contracts until the audit Criticals/Highs in `§13.1`/`§11.1` are implemented (`_disableInitializers`, per-record-type scoping `isWhitelistedFor`, `issuedBy` originator binding, admin-only `burn`, `AccessControlDefaultAdminRules` + multisig admin). Do **not** expose custody endpoints publicly — `/admin/*`, localhost/session-bound, `/unlock` rate-limited (`§11.4`).
3. **Contextual four-fragment verification.** Credential validity = the **three authenticity pillars** (integrity **and** on-chain status **and** DNS-identity-cross-checked-against-registry). The **`ownership`** fragment (`ownerOf(dogTagId)==userWalletAddress`) is **contextual**: it gates only the owner's *self-import*, and is `NOT_APPLICABLE` (never INVALID) for third-party verifiers — making it universally required breaks the groomer/airline/vet import flows. Fragments are 4-state `VALID|INVALID|ERROR|NOT_APPLICABLE`. Integrity alone means nothing. Code **`§11.3`** (not the superseded §1.7).
4. **Test as you build.** Each phase ships with tests and a green build before the next phase starts. No phase is "done" without its acceptance criteria met.
5. **Self-hosted reality.** Treat each business backend as an untrusted-to-others, operator-controlled deployment. Trust comes from the **central registry** + on-chain whitelist, not from the deployment claiming things.
6. **Use the pinned stacks:** Foundry + OZ v5 (`evm_version = paris`); Rust + Axum + MongoDB + Alloy (not ethers-rs); React + Vite + TS + Tailwind + shadcn; Kotlin/Compose + Swift/SwiftUI; shared verify via UniFFI. Uncommon ports: admin 39741/39742, vet 41873/41874, groomer 43617/43618; Mongo internal-only.
7. **Privacy — nothing personal on-chain, ever.** On-chain holds only **salted commitments** (per-field random 16-byte salts; salts off-chain), revocation/status, timestamps, and **non-personal refs** (DIDs, accreditation refs). A salted hash is still personal data and an unsalted hash of a low-entropy microchip number is brute-forceable — salting is the privacy mechanism, not just anti-forgery. All `Owner` PII and Art. 9 service-attestation data stay off-chain, encrypted, deletable. **Right-to-erasure = delete the off-chain record + destroy its salt/key** (renders the on-chain commitment unlinkable; documented as a mitigation, not a regulator-blessed safe harbour). **Mandatory DPIA.** CCPA/GDPR 45-day delete runs the **same** erasure flow.
8. **Neither signing mode can fake issuance.** Both `WalletStrategy` and `BackendStrategy` build the Merkle root / wrapped doc **server-side (shared SDK) — identical in both modes**; the backend **re-verifies on-chain** (`RootIssued` event + `issuedAt[root] != 0`) before marking a record issued. A lying/buggy frontend cannot fake issuance.
9. **On-chain proof-of-verification rules (v3/v4).** There is a **SINGLE Poseidon root `R`** — the circuit proves the same `R` that is anchored on-chain by `issue(R)`, and the `VerificationRegistry` checks `DogTagIssuer.isValid(R)` **directly** on both paths. (No `rKec`/`rZk` duality, no `zkCommit`/`kecOf`/`zkIndex` — those are deleted.) The EIP-712 **`VerificationConsent`** binds the **`relayer`** (verifier wallet) into the signed struct, and the contract enforces `msg.sender == relayer` on both paths (plain relayer pattern — **no EIP-2771/ERC-4337**). The credential is **purpose-bound** (purpose-scoped `VERIFY:` whitelist; `purpose` bound into the nullifier) and **subject-bound** (`keyOf[subject]==keyHash` where `keyHash=Poseidon(Ax,Ay)`; `ownerOf(dogTagId)==subject`; `subject` bound into the EdDSA message). The shared Poseidon **`nullifier = Poseidon(DS_NULLIFIER, dogTagId, purpose, relayer, subject, nonce)` is a public signal and shared across both paths** via a single `consumed` set (one consent = one attestation; **never derive it from proof bytes** — snarkjs #383). The registry **range-checks ALL public signals** against the BN254 scalar field (#358). **ZK is the data-minimization default for sensitive purposes** (normal path is the fallback when an on-chain `credentialRoot` commitment is genuinely required).

## Phased plan (build in this order — matches `implementation.md §10`)

### Phase 1 — The trust core: shared Poseidon standard SDKs
**Goal:** bit-identical Poseidon roots across FOUR languages (circom, TS, Rust, Solidity).
- **Lock the FINALIZED schemas first (before coding validators):** coded vaccine (`vaccineProductCode` = USDA APHIS PCN + separate `vaccineManufacturer`), VBO breed (`breedVbo` + `breedLabel`), `microchip` **object** (15-char string, never a float), trust-tiered **service attestation** (`assistanceType`/`issuerTrustTier`/`legalContext`, Art. 9 off-chain), **VC 2.0 envelope** (array `@context`/`type`, prose→`description`, `credentialStatus`/`id`/`credentialSchema`), and first-class **`Owner`** entity (off-chain PII). Use canonical names/enums from CHANGESPEC §0.
- **Pin the Poseidon trust core FIRST:** ONE circomlib BN254 parameter set across circom / `poseidon-lite` (TS) / `light-poseidon` (Rust, `new_circom(n)`) / `poseidon-solidity` (`PoseidonT3..T7`). Implement **byte→field packing** `fieldOf`/`bytesToField`: prefix an 8-byte big-endian length, split into 31-byte big-endian limbs (each `< 2^248 < p`), then domain-sep fold `acc = DS_BYTES; for L: acc = Poseidon(acc, fieldFromLimb(L))`; scalars (15-digit chip, timestamps, `typeTag`, addresses as uint160) reduce into `[0,p)` as one field. Domain tags `DS_LEAF=1, DS_NODE=2, DS_BYTES=3, DS_NULLIFIER=4` are used as the **first input slot** (not capacity IV) to stay on the exact circomlib API in all 4 libs.
- Implement `crates/dogtag-standard-rs` and `packages/dogtag-standard-ts`: `encodeValue` (**unchanged** — pinned decimal/integer grammar, `assertNotFloat`, NFC, typed input, first-two-colons parse), `hashLeaf = Poseidon(DS_LEAF, fieldOf(keyPath), fieldOf(salt16), fieldOf(typeTag), fieldOf(value))` (5 inputs → width t=6), `buildMerkle` with `hashNode = Poseidon(DS_NODE, min(a,b), max(a,b))` (commutative sorted-pair compared as integers in `[0,p)`, odd-promotion, single-leaf root), `merkleProof`/`processProof`, `flatten`/`unflatten` (pinned grammar), `wrapDocument` (typed input, returns a **single root `R`** — no `hashLeafZk`/`rZk`), `obfuscate`, `verify` (3-pillar, tri-state).
- **v3:** add the EIP-712 **`VerificationConsent`** struct (`{dogTagId, recordType, credentialRoot, relayer, subject, nonce, deadline}`, domain `{name:"DogTag", version:"1", chainId:135, verifyingContract: VerificationRegistry}`) to the shared `consent` module (UniFFI-exported in both SDKs). Consent signs over the single root `R`.
- Author `testvectors.json` (leaf hashes, roots, proofs; include decimal `22.7`/`0.5`, timestamps with `:`, nested/array/empty, single-leaf, odd sizes 1–9, obfuscation). **v4 parity vectors (gating):** the anchor `poseidon([1,2]) = 0x115cc0f5...189a`, plus `fieldOf`/leaf/Merkle-root/`nullifier = Poseidon(DS_NULLIFIER, dogTagId, purpose, relayer, subject, nonce)` vectors, all asserted bit-identical across **circom + TS + Rust + Solidity**.
- Add a Solidity node-hash test proving on-chain Poseidon agreement at the node level.
**Acceptance:** the **CI Poseidon anchor-vector gate** — `poseidon([1,2]) = 0x115cc0f5...189a` — passes bit-identically in **all four** (circom, TS, Rust, Solidity) before anything depends on the SDKs; the anchor/leaf/Merkle/nullifier parity vectors match across all four; a value cannot be swapped/removed while keeping the root; obfuscation preserves the root.

### Phase 2.5 — ZK verification subsystem
**Goal:** a consent-bound on-chain proof-of-verification, both normal and Groth16 ZK paths.
- Build **`circuits/`**: a circom **Poseidon-Merkle + EdDSA-BabyJubjub consent** circuit on **BN254 (Groth16), ~12–18k constraints, sub-second proving**, on the **same pinned circomlib Poseidon** as the SDKs. Public signals (order): `[dogTagId, purpose, relayer, subject, nullifier, keyHash, R]` — the proven root `R` IS the actual issued root (**no `rZk`, no dual root, no `zkIndex`/`cloneOf`/`issuerForAny`**). Private: leaf values/salts/typeTags/keyPath-hashes, the in-circuit ordered (sortPair+mux) Poseidon Merkle path, `consentNonce`, the EdDSA-BabyJubjub signature + consent pubkey. Proves: private leaves → the proven root == public `R`; the `dogTagId` leaf == public `dogTagId`; `keyHash == Poseidon(Ax,Ay)`; the consent key signed a message binding `(dogTagId, purpose, relayer, R, subject, nonce)`; `nullifier == Poseidon(DS_NULLIFIER, dogTagId, purpose, relayer, subject, nonce)`. The circuit does **NOT** prove `isValid` (the registry re-checks `isValid(R)` on-chain directly).
- **Trusted setup:** reuse Hermez/Perpetual Powers of Tau (phase 1) + a **multi-party phase 2 (≥3 contributors) ending in a public random beacon**; publish the transcript, **pin the `.zkey` hash**, ship it in the prover image. Generate `Groth16Verifier` via snarkjs `zkey export solidityverifier`.
- Implement **`VerificationRegistry`** (custom — **not** EAS) with `recordVerification(consent, userSig)` (normal: `ECDSA.recover == subject`, **`isValid(R)` directly**) and `recordVerificationZK(a, b, c, pub[7])` (ZK: `verifyProof`, **`isValid(R)` directly** on the public root `R`, require `keyHash == ConsentKeyRegistry.keyOf[subject]`, `ownerOf(dogTagId)==subject`); shared `consumed` nullifier set; `msg.sender == relayer` enforced both paths; **range-check ALL public signals** against the BN254 field; **purpose-scoped `VERIFY:`-namespace whitelist** gating (separate from issuer roles); `Verified(dogTagId, relayer, subject, purpose, nullifier, ts)` event. Implement **`ConsentKeyRegistry`** (`bindConsentKey(babyJubPubKey, ecdsaSig)` → `ecrecover` binds the BabyJubjub key to the secp256k1 `userWallet`, exposing `keyOf[subject]`). Normal-path nullifier `Poseidon(DS_NULLIFIER, dogTagId, purpose, relayer, subject, nonce)` via `poseidon-solidity` `PoseidonT7` (same pinned instantiation as the circuit). `setZkVerifier` behind a real timelock.
- Foundry tests covering **both paths**, **shared-nullifier double-spend revert** (one consent across both paths cannot record twice), **range-checks** (BN254-field gating), **relayer binding** (`msg.sender != relayer` reverts), **subject↔key binding** (`keyHash != keyOf[subject]` reverts), **`ownerOf(dogTagId)==subject`**, and **purpose binding** (wrong-purpose whitelist/nullifier reverts).
**Acceptance:** a valid proof verifies on-chain and emits `Verified` with the registry checking `isValid(R)` directly; a replayed proof, a cross-relayer submission, a malleated proof, a wrong-subject-key, a wrong-owner, and a wrong-purpose all revert; out-of-field public signals revert.
