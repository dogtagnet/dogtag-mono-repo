## 1. CROSS-DOC CONSISTENCY findings

### C-1 (High) — research/12 uses a stale, INCONSISTENT name + signal set throughout; it contradicts CHANGESPEC §0
`research/12-verification-integration.md` is cited as a source by `architecture.md §4.7`, `implementation.md §3.9/§11.8`, and `§13.7`, but it predates CHANGESPEC-v3 §0 and still uses the **superseded** vocabulary everywhere:
- Event name **`VerificationAttested`** (§1.2 step 5/6, §3 table, §4.1/§4.2 maps, §6 Phase-8) vs the canonical **`Verified`**.
- Contract method **`recordVerificationZk`** (lowercase `k`) vs canonical **`recordVerificationZK`**.
- Consent struct fields **`merkleRoot`, `verifier`, `userWallet`, `purpose`, `challenge`** (§1.2 step 3) vs the canonical 7-field `VerificationConsent {dogTagId, recordType, credentialRoot, relayer, subject, nonce, deadline}`. `verifier`→`relayer`, `userWallet`→`subject`, `merkleRoot`→`credentialRoot` are all renamed in §0 but research/12 never updated.
- **3-signal public order `(dogTagId, groomerRelayer, userWallet)`** (§1.2 step 6, §2.1, §3 components table, §6 Phase-2.5) vs the canonical **5-signal `[dogTagId, relayer, subject, nullifier, rZk]`**. This is the "leftover sketch" the brief asked to hunt: research/12 has **no `nullifier` and no `rZk` in its public-signal list** anywhere. A coding agent reading 12 as a source would build a 3-signal circuit with no nullifier — directly defeating the shared-nullifier anti-double-spend guarantee.
- References **`CHANGESPEC-v2`** (§Normative posture line 5, §4.3 heading) — the change-spec is now **v3** (`CHANGESPEC-v3.md`).
- Self-import mode string: research/12 §1.1/§1.2 uses `mode:"self-import"` correctly; but its consent JWT `aud:"dogtag-verify"` and `scope:"present:credential"` (§1.2 step 2) **conflict** with the canonical `aud:"dogtag-mobile"` used in `implementation.md §3.9/§4.1/§11.8(g)`. The session QR path also differs: `/v?t=&s=<sessionId>` (12) vs `/v?t=` carrying `sub:sessionId` in-JWT (impl).

**Fix:** Add a banner to the TOP of `research/12` (like impl §1.7's SUPERSEDED banner): "Superseded by CHANGESPEC-v3 §0 + arch §4.7 + impl §11.8 for all names, the consent struct, the 5-signal public order `[dogTagId,relayer,subject,nullifier,rZk]`, the event name `Verified`, `recordVerificationZK`, and JWT `aud:dogtag-mobile`. Read this brief for rationale only." Otherwise an agent that "reads the research" (BUILD_PROMPT First-action #1) will mis-build.

### C-2 (High) — `purpose` vs `recordType` collision is unresolved and self-contradictory across the canonical docs
The canonical docs themselves disagree on what `purpose` is and where it comes from:
- CHANGESPEC §0 + §9 define `purpose = keccak256(label)` with labels `GROOMING_INTAKE / TRAVEL_PRESENTATION / AIRLINE_CHECKIN / VET_INTAKE`, **distinct** from `recordType` (the credential type, e.g. `keccak256("VACCINATION")`).
- But the **`VerificationConsent` struct has a `recordType` field and NO `purpose` field** (CHANGESPEC §0, arch §3.6, impl §1.10/§11.8). And `VerificationRegistry.recordVerification` sets `bytes32 purpose = c.recordType;` (impl §11.8(a) line 1412; CHANGESPEC §2 line 51 uses bare `purpose` undefined). So on the **normal path, `purpose` IS `recordType`** — the grooming/travel/airline labels are never actually used on-chain.
- The **`VERIFY:` whitelist key** is `keccak256("VERIFY:"||purpose)` and `purpose==recordType` on the normal path, so a groomer is whitelisted for `VERIFY:VACCINATION`, **not** `VERIFY:GROOMING_INTAKE`. This contradicts CHANGESPEC §0's stated purpose enum and arch §3.6's "keyed by `purpose`...labels such as GROOMING_INTAKE".
- The `/verify/session/start` endpoint takes BOTH `{purpose, recordType}` (impl §3.9 line 587, §11.8(g)) but the consent only carries `recordType`, so the operator-chosen `purpose` is **dropped before it reaches the consent/registry** on the normal path.

**This is the brief's "how is `purpose` chosen/validated" gap.** Net: the docs simultaneously claim `purpose` is a first-class presentation label AND set it equal to `recordType`. Pick one:
- **Option A (recommended):** add `bytes32 purpose` to `VerificationConsent` as a distinct field, sign it, emit it; gate `VERIFY:purpose`; keep `recordType` for the `isValid` issuer lookup. This is the only way the GROOMING_INTAKE enum is real and the `Verified.purpose` event field is meaningful.
- **Option B:** delete the `purpose` enum from CHANGESPEC §0/§9 and arch §3.6, state plainly "purpose == recordType in v1," and rename the event field/`VERIFY:` namespace to `recordType`. (But then Phase-8's "no Art.9 purpose on-chain" rule loses its hook — see P-3.)

### C-3 (High) — ZK path cannot resolve `purpose`/`recordType` for the `VERIFY:` whitelist; it hardcodes `bytes32(0)`
On the ZK path, `recordType`/`purpose` are deliberately private (not public signals — only `[dogTagId,relayer,subject,nullifier,rZk]`). So `recordVerificationZK` whitelists against `keccak256("VERIFY:"||bytes32(0))` (impl §11.8(a) line 1434 — explicit `bytes32(0)`, comment "purpose private in ZK"). Consequences, none of which any doc acknowledges as a resolved design:
1. **The `VERIFY:` per-purpose gating is meaningless on the ZK path** — every ZK relayer is gated by the SAME `VERIFY:0x00...0` key, so admin can only grant "may ZK-verify anything" not "may verify GROOMING_INTAKE." The separate-from-issuer-roles property survives, but per-purpose scoping (CHANGESPEC §0, arch §4.3) does **not**.
2. `issuerForAny()` (impl §11.8(a) lines 1440/1445) is a hand-wave: the comment says "either a single protocol issuer or per-circuit recordType pinned in the verifying key." But there are **many** `DogTagIssuer` clones (one per recordType per business — arch §4.4), and `kecOf[rZk]` lives on the specific clone that issued. A single `issuerForAny()` cannot find the right clone without knowing recordType. This is **under-specified to the point of un-buildable** for the ZK path.

**Fix:** Make `recordType` (or `purpose`) a **public signal** on the ZK path too (it's a non-personal label keccak, not PII — and §P-3 already says service-animal purposes must NOT be on-chain in clear, which the ZK-keeps-it-private story relies on; reconcile). With recordType public, the registry resolves `issuerFor[recordType].kecOf[rZk]` and gates `VERIFY:purpose` correctly. If recordType must stay private for Art.9 reasons, then specify a concrete `issuerFor`-by-`rZk` reverse index (e.g. a registry-level `rZk → clone` map written by `zkCommit`), and document that ZK per-purpose whitelisting is intentionally coarse.

### C-4 (Medium) — `Verified` ZK-path emits `purpose=bytes32(0)`, but arch §11.1/§13.7 and CHANGESPEC §5 describe the event as carrying the `(dogTagId, relayer, subject, purpose)` tuple
impl §11.8(a) line 1443 emits `Verified(..., bytes32(0) /*purpose*/, nf, ts)` on the ZK path. arch §13.7 mitigation (1) says "the ZK path...only the tuple `(dogTagId, relayer, subject, purpose)` + nullifier are emitted." With `purpose=0` the tuple has no purpose. Minor but it makes §13.7's "(dogTagId,relayer,subject,purpose)" description inaccurate for ZK. Either drop `purpose` from the §13.7 ZK-tuple description or make recordType/purpose public (ties to C-3).

### C-5 (Medium) — `IssuerRegistry` body shown in `implementation.md §2.1` is the OLD single-boolean version; contradicts §11.1 + arch §4.3 + §13.1 C-2
impl §2.1 (lines 320-328) still shows `IssuerRegistry` with `mapping(address=>bool) _whitelisted`, `whitelistIssuer/delistIssuer/isWhitelisted(address)` — the pre-audit global boolean. The normative §11.1 (lines 1030-1043) replaces it with per-recordType `_wl[rt][s]` + `whitelistFor/delistFor/isWhitelistedFor`. §2.2's `DogTagIssuer.onlyWhitelisted` even calls `registry.isWhitelisted(msg.sender)` (single-arg, §2.2 line 337) while §11.1 and the whole VERIFY: design require `isWhitelistedFor(recordType, signer)`. Also impl §2.5 Deploy step 6 calls `registry.whitelistIssuer(...)` and §5.3 admin portal "triggers on-chain `whitelistIssuer`" — both stale.
**Fix:** §2.1/§2.2/§2.4 are pre-remediation sketches; §11.1 is normative. Either delete the stale bodies or add a one-line "SUPERSEDED by §11.1" banner on §2.1-§2.4 (consistent with how §1.7 is handled). At minimum fix §2.5 step 6 + §5.3 to `whitelistFor(recordType, addr)`.

### C-6 (Medium) — `Deploy.s.sol` (impl §2.5) doesn't deploy the v3 contracts in dependency order / references a missing arg
impl §2.5 lines 431-434 add steps 7-9 (Groth16Verifier, ConsentKeyRegistry, VerificationRegistry, `whitelistFor("VERIFY:"||purpose, relayer)`). But: (a) the `VerificationRegistry` constructor in §11.8(a) takes `(ir, sbt, zk, ck, admin)` and needs `setIssuerFor(recordType, clone)` calls afterward — step 9 never wires `issuerFor[]`, so every `recordVerification` reverts `"no issuer"`. (b) Step 5 creates issuer clones via the **un-permissioned** `createIssuer(name, recordType, salt)` (§2.3) but §11.1's hardened factory is `createIssuer(name, recordType, business) onlyRole(ADMIN)` with `salt=keccak256(recordType,business)` — different signature. The deploy script uses the stale signature.
**Fix:** Add a deploy step `registry.setIssuerFor(keccak256(label), cloneAddr)` for each issuer clone; align `createIssuer` args with §11.1.

### C-7 (Low) — `recordType` non-obfuscatable list vs ZK "recordType private"
impl §11.7(d) requiredPaths (non-obfuscatable) and §11.3 list `@context[*]`, `type[*]`, `credentialStatus.*`, `credentialSubject.dogTagId`. Fine. No contradiction, but note the ZK circuit proves only the `dogTagId` leaf equals public `dogTagId` (CHANGESPEC §3(b)); it does **not** prove `recordType`, `credentialStatus`, or `@context` membership. So a ZK attestation does not assure the verifier the credential is even a vaccination cert (vs any leaf set committed to `rZk`). Acknowledge this scope limit in arch §4.7 — currently it reads as if ZK proves a full valid credential.

### C-8 (Low) — nullifier formula uses `uint160(address)` casts in the normal path; ensure the circuit packs `relayer`/`subject` identically
impl §11.8(a) line 1423 computes `Poseidon.hash4(c.dogTagId, uint160(c.relayer), uint160(c.subject), c.nonce)`. The circuit (§11.8(d)) computes `Poseidon4(dogTagId, relayer, subject, consentNonce)` with `relayer`/`subject` as "address as field element." These must be the IDENTICAL field encoding or the shared `consumed` set splits (normal nf != ZK nf for the same logical consent → double-record possible). This is the crux of "one consent = one attestation across both paths." The docs assert the shared set but never pin the address→field encoding. **Fix:** state normatively in §11.8 that addresses enter Poseidon as their `uint160` value (no abi-pack, no checksum string). (Flag for the ZK-math auditor too — borderline in-scope.)

## 2. FLOW SOUNDNESS findings (session → consent → relay → proof → registry)

### F-1 (Critical) — Replay/forgery across the relay hop: nothing binds the consent signature to the session `challenge`
The session JWT carries a one-time `challenge` (impl §3.9 line 591, §11.8(g); CHANGESPEC §1 step 1). But the **`VerificationConsent` struct has no `challenge` field** (canonical 7 fields). The signature therefore does NOT cover the challenge. The challenge is checked only at `/verify/consent/submit` against `verify_sessions[sessionId]` server-side state (impl §3.9 require `consent.relayer==s.relayer`; note it does **not** even check the challenge — research/12 §1.2 step 4 did check `sessions[...].challenge == consent.challenge`, but that field was dropped in the canonical struct). Consequences:
1. The on-chain `recordVerification(consent, userSig)` enforces only `deadline`, `relayer==msg.sender`, `ECDSA.recover==subject`, `ownerOf==subject`, `isValid`, and `!consumed[nf]`. The session/challenge is **purely an off-chain UX nicety** — it never reaches the chain.
2. **The actual on-chain replay defense is the `nonce`/`nullifier`, NOT the challenge.** A signed consent is reusable by the relayer until `deadline` for any `(dogTagId, relayer, subject, nonce)` not yet consumed. Since the same `nonce` yields the same `nullifier`, a relayer cannot double-record the same consent — good. But a **malicious or compromised verifier backend** that receives the relayed `{consent, sig}` can submit it on-chain **without** ever showing the user a session (the user signed `relayer=thatVerifier` once; the verifier can record a `Verified` event at a time of its choosing within `deadline`). The user's intent ("verify for THIS grooming visit now") is not cryptographically bound — only "I consent to relayer X recording one verification of dogTagId/root with nonce N before deadline."

**Assessment:** This is acceptable-by-design IF documented (the trust model is: the user authorizes relayer X to record one attestation; the relayer is whitelisted and bound as msg.sender). But it is **not** what arch/§13.7 imply ("consent-gated, purpose-bound"). The `purpose` isn't signed (C-2), the `challenge` isn't signed, so the on-chain record is **purpose-agnostic and session-agnostic**. A whitelisted relayer can mint a `Verified(user, pet, relayer, ts)` for any pet whose owner ever consented to it once, within deadline.
**Fix:** Either (a) add `challenge` (or `sessionId`) AND `purpose` to the signed `VerificationConsent` struct so user intent is bound on-chain, or (b) explicitly document in §13.7 that the on-chain attestation binds only `(relayer, subject, pet, root, nonce, deadline)` and that purpose/session are off-chain context — and shorten `deadline` (currently `now+15m`, mobile §6.6 line 947) accordingly. Recommend (a): add `purpose` + `nonce`-as-challenge.

### F-2 (High) — The central→verifier relay hop is UNAUTHENTICATED (no HMAC); contradicts the system's own cross-backend trust model
`POST /v1/verify/consent` (central) relays to `POST verifierApiBase + "/verify/consent/submit"` (impl §4.1 line 720; CHANGESPEC §1 step 2). Every OTHER central→business call is **HMAC-signed** (appointments §8.3, erasure propagation §11.6, `appointment-events` §11.4 C-2 resolves HMAC-key-by-path). But:
- `/verify/consent/submit` (impl §3.9 line 597, §11.8(g)) has **no auth requirement listed** — not operator session (it's machine-to-machine), not HMAC, not the session JWT. It only checks `verify_sessions[sessionId]` exists and `consent.relayer==s.relayer`.
- So **any party on the network can POST a `{sessionId, consent, sig}` to a verifier's `/verify/consent/submit`** if they know a pending sessionId. The consent sig still must recover to `subject` and the on-chain checks still hold, so they can't forge a record for a non-consenting user — BUT they can (a) drive the verifier to spend PLASMA gas submitting a (replayed, still-valid-nonce) consent, and (b) front-run/grief the legitimate flow.
- More importantly the **verifier backend trusts `sessionId` + `consent` arriving from "central" with no proof it came from central.** The discovery doc (`businesses{...hmacKeyId}`) already establishes a shared HMAC secret per business — the relay should use it.

**Fix:** HMAC-sign the central→verifier `/verify/consent/submit` relay with the per-business key (same mechanism as `/v1/businesses/{bid}/appointment-events` and `/v1/businesses/{bid}/erase`). Add to impl §3.9/§11.8(g): "verify_hmac(req, hmacKeyFor(businessId))". Add to BUILD_PROMPT Phase-3/Phase-4 acceptance. The brief explicitly asked "HMAC for the central→verifier relay?" — **answer: it is missing and must be added.**

### F-3 (High) — `/verify/consent/submit` never re-verifies the EIP-712/EdDSA signature before building the proof / submitting
On the normal path, the verifier backend builds `prepared = tx("recordVerification", consent, sig)` and submits; the on-chain `ECDSA.recover==subject` is the only sig check. On the ZK path, the prover takes `eddsaSig:sig` and builds the proof — if the EdDSA sig is invalid the circuit fails (good). But:
- The submit handler should **verify `consent.recordType == keccak256(s.recordType)`** (impl §3.9 line 599 does this for normal, but §11.8(g) drops it) and **verify the ECDSA consent sig off-chain** before spending gas (normal path), to fail fast rather than revert on-chain (consistent with the §11.6 `preflightWhitelist` "fail fast — wallet mode user pays gas" philosophy). Currently a bad normal-path sig wastes a full tx + revert.
- research/12 §1.2 step 4 DID specify "verify EIP-712 consentSig recovers to consent.userWallet" + "require consent.verifier == activeSigner()" + one-time challenge at submit. The canonical impl §11.8(g) **lost all three of these checks**. This is a regression from the research into the normative code.
**Fix:** Re-add to §11.8(g) `/verify/consent/submit`: off-chain `recover(consent)==subject` (normal), `consent.relayer == activeSigner()`, `consent.recordType == keccak256(s.recordType)`, and consume the session (one-time). These are cheap fail-fast guards.

### F-4 (Medium) — Two different one-time-JWT audiences/jti consumers; double-consume risk on the relay
The session JWT (`aud:"dogtag-mobile"`, `jti`) is consumed at central `/v1/verify/consent` (`consume_jti(claims.jti)` — impl §4.1 line 712). Good — one consent submission per session. But the verifier's `/verify/consent/submit` keys off `sessionId` (`verify_sessions[sessionId]` status `pending→recorded`), and there's no atomic guard described (cf. §11.4 `consume_jti` atomic insert). Two concurrent relays (or the F-2 unauthenticated replay) could both pass `status=="pending"` before either flips it to `recorded`. **Fix:** make the verifier session-status transition atomic (compare-and-set `pending→recording`), mirror §13.3's "`jti` one-time use is atomic."

### F-5 (Medium) — `ConsentKeyRegistry.bindConsentKey` is one-time and irreversible; no rebind path on MPC key rotation / `recover()`
`keyOf[wallet]` is set once and `require(keyOf[msg.sender]==bytes32(0))` blocks rebinding (impl §11.8(b)). But the SBT owner address can change via `recover()` (§11.7(a)) — lost-key re-bind to a NEW owner address. After recovery the new `subject` (= new `ownerOf`) has **no bound BabyJubjub key**, so the ZK path requires a fresh `bindConsentKey` from the new wallet (fine), but more subtly: a consent key bound to the OLD address is now orphaned, and the §11.8(a) ZK check (subject↔key via `keyOf[subject]`) uses the CURRENT `ownerOf`. No correctness bug, but the "bind once, ever" with no rebind/rotate is under-specified for the recovery + MPC-rotation lifecycle. **Fix:** allow re-bind gated by current control (e.g. `bindConsentKey` callable again if a newer EIP-712 from `msg.sender` supersedes), or document that post-recovery the new owner must bind a fresh key and the ZK path is unavailable until they do.

### F-6 (Low) — Mobile `nextConsentNonce` source unspecified; nonce collision → silent double-spend block
mobile §6.6 line 945 `nonce = nextConsentNonce(claims.relayer, dogTagId)` — but there's no on-chain `nonces[subject]` mapping (unlike the SBT `recoverNonce`). The nullifier `Poseidon(dogTagId,relayer,subject,nonce)` is the only replay guard; if the device reuses a nonce (reinstall, multi-device), the second legitimate verification by the same relayer for the same pet silently reverts `"replayed"`. **Fix:** spec the nonce as a monotonic per-(relayer,dogTagId) counter persisted in the device wallet, or random 256-bit (collision-negligible). State it in §1.10/§6.6.

### F-7 (Low) — `restrictToWhitelistedRelayers` admin toggle defaults true but can be flipped off, enabling anonymous attestations
arch §4.7 / impl §11.8(a) ship `restrictToWhitelistedRelayers=true` with an admin `setRelayerRestriction(bool)`. If flipped off, ANY address can record `Verified` events (subject sig still required). Acceptable as an admin escape hatch, but it should be DPIA-noted (an unwhitelisted relayer recording behavioral linkage) and BUILD_PROMPT should test the toggle. Currently undocumented in §13.7. **Fix:** note in §13.7 that disabling the restriction widens who can anchor behavioral data.

## 4. PRIVACY / GDPR (research 07) — verdict on the on-chain verification linkage

### P-1 (High, but acknowledged) — the permanent behavioral linkage IS personal data and IS acknowledged; the acknowledgment is internally consistent
The on-chain `Verified(dogTagId, relayer, subject, purpose, nullifier, ts)` is a permanent record of *which user (subject wallet) had which pet verified by which business, when.* Against research/07 §4.1 (Recital 26 linkability), §4.2 (a commitment can be personal data), §4.5 (CCPA broad PI), this is **pseudonymous personal data on an immutable, globally-replicated ledger** — and it is a **behavioral time-series**, more sensitive than the static `ownerOf↔wallet` link (research/12 §5.1 correctly calls this an amplification from a static binding to a behavioral graph).
**Acknowledgment check — PASS:** arch §11.1 final bullet, arch §13.7 (all 5 mitigations), CHANGESPEC §5, impl §4.5/§11.6 erase() NB-comments, and BUILD_PROMPT Phase-8 behavioral-privacy gate all acknowledge it and put it in DPIA scope. This is consistent across the canonical docs.

### P-2 (Critical) — Consent RECEIPTS are in erasure scope, but the docs do not consistently put the on-chain consent BINDING / `ConsentKeyRegistry` linkage in DPIA scope, and the relay leaves a PII-bearing consent copy at the verifier
- **Erasure scope:** `verification_records` + off-chain `VerificationConsent` copies + consent receipts ARE in the crypto-shred scope (arch §9.1/§9.2, §13.7 mitigation 4, impl §4.5 erase() lines 779-783, §11.6 erase()). **PASS** on central + business backends.
- **GAP:** the verifier backend stores `verification_records` (impl §9.2) AND receives the relayed `{consent, sig}` (which contains `dogTagId, subject, recordType, nonce` — pseudonymous PII). The **cross-backend erasure propagation** (§11.6 `propagate_erasure` central→every business) is specified for the issuer/record controller relationship, but it is **not explicitly stated that the VERIFIER businesses (which may be different from the issuer of the credential) are also in the propagation set.** A groomer who verified a pet holds a `verification_records` row + consent copy; on the owner's erasure request, does central know to propagate to that groomer? The `businesses` registry + `verification_records.relayer` give the address, but §11.6 propagates to "EVERY business backend holding copies" without a concrete index from `ownerId → verifier businesses`. **Fix:** specify that erase() enumerates `verification_records{ownerId}` → distinct `relayer` → resolve business via registry → `POST /v1/businesses/{bid}/erase`. Add to BUILD_PROMPT Phase-4 acceptance ("erasure reaches verifier businesses, not just issuer businesses").
- **`ConsentKeyRegistry.keyOf[wallet]` is on-chain `Poseidon(babyJubPubKey)` keyed by `userWallet`** — another permanent on-chain wallet-linked datum (a second pseudonymous identifier bound to the subject address). It is **not mentioned anywhere in §11.1/§13.7/DPIA scope.** It's low-sensitivity (a pubkey hash) but it's immutable on-chain wallet-linked data. **Fix:** add `ConsentKeyRegistry.keyOf` to the §13.7 DPIA enumeration as residual on-chain pseudonymous data (cannot be erased; fresh-per-pet address bounds it, but note the key may be shared across a person's pets if derived once — see P-5).

### P-3 (High) — `purpose` on-chain Art. 9 leak: the mitigation is stated but BROKEN by the normal-path `purpose==recordType` reality (ties to C-2)
research/12 §5.2 sets the hard rule: do NOT put service-animal/disability `purpose` labels on-chain in cleartext; use ZK or a non-revealing hash for Art.9-adjacent verifications. arch §13.7 / CHANGESPEC §5 echo "ZK is the default for sensitive purposes."
**But:** on the normal path `purpose == recordType` (C-2), and `recordType` for a service attestation is `keccak256("SERVICE_ATTESTATION")` — which is emitted as `Verified.purpose` in cleartext keccak. A keccak of a known short label is **trivially reversible by dictionary** (there are ~7 record types). So a **normal-path** verification of a `SERVICE_ATTESTATION` credential would emit a publicly-recognizable Art. 9 marker on-chain. The "use ZK for Art.9" rule is only a *default*, not *enforced* — nothing in the registry or backend BLOCKS a normal-path `SERVICE_ATTESTATION` verification.
**Compounding:** arch §3.6/§13.4 already say `SERVICE_ATTESTATION` data is "off-chain only, NEVER hashed on-chain" — but that's about the credential *content*; the **verification event** of that record type re-introduces the category on-chain via `recordType`/`purpose`. Also note `SERVICE_ATTESTATION` has **no on-chain issued root** in the issuance model (it's off-chain only, no `DogTagIssuer.issue`), so `recordVerification`'s `isValid(credentialRoot)` would fail anyway for normal mode — meaning a service-attestation verification is *implicitly* impossible on the normal path. This is a latent inconsistency: the verification flow assumes an on-chain `isValid` root exists, but the most privacy-sensitive record type deliberately has none.
**Fix:** (a) Make it NORMATIVE (not default) that any Art. 9-adjacent `recordType`/`purpose` MUST use the ZK path AND must not be emitted in `Verified.purpose` (emit `bytes32(0)` or omit). (b) Explicitly state whether `SERVICE_ATTESTATION` is verifiable at all (it has no on-chain root) — if yes, ZK-only over its `rZk`; if no, say so. (c) Add a backend + registry guard rejecting cleartext Art.9 purposes (a deny-list, since the label set is tiny). This is the single most regulator-fatal gap if shipped as-is.

### P-4 (Medium) — `subject`-on-chain in v1 IS a residual that the DPIA must document; it is documented, but the v1-vs-v2 status needs sharpening
arch §13.7 mitigation 3 ("ZK-v2: publish nullifier INSTEAD of subject") + CHANGESPEC §7 decision-point 3 + research/12 §5.2 correctly frame **v1 publishes `subject` (the wallet) on-chain** as the residual, and v2 (nullifier-instead-of-subject) as the privacy endgame. This is consistent and the residual is acknowledged.
**Sharpen:** (a) The `Verified` event indexes `subject` (impl §11.8(a) line 1400 `address indexed subject`) — being an indexed topic makes it **maximally enumerable** (filter all events by a wallet). The v1 residual is therefore worse than "present in calldata." Note this explicitly in §13.7. (b) Mitigation 3's claim that emitting nullifier-instead-of-subject "severs the on-chain user-address link entirely" is only true if the **registry no longer checks `sbt.ownerOf(c.dogTagId)==c.subject` on-chain** — but the normal path REQUIRES that check (`subject !owner` revert), so the normal path can never drop `subject`. Mitigation 3 is **ZK-only** by construction. State that v2-privacy-endgame is ZK-path-only; the normal path structurally exposes `subject`. (c) The brief's question "is `subject`-on-chain in v1 a residual that must be DPIA-documented?" — **YES, and it is documented; tighten per (a)/(b).**

### P-5 (Medium) — fresh-per-pet `subject` address: the BabyJubjub consent key may be derived ONCE and shared across pets, defeating the per-pet unlinkability
The privacy story leans on "fresh per-pet derived address as `subject` bounds linkage to one pet" (arch §13.7 mitigation 2, §11.1, BUILD_PROMPT Phase-8). But:
- The SBT is minted to `m/44'/60'/0'/0/{petIndex}` (a fresh per-pet secp256k1 address) — good.
- The **BabyJubjub consent key** is `deriveBabyjubConsentKey(seed)` (impl §1.10 line 310, §6.6 line 931) — derived from the seed, "deterministic, distinct domain from the secp256k1 path." It is **not stated to be per-pet.** If one BabyJubjub key is bound (via `ConsentKeyRegistry.bindConsentKey`, keyed by `userWallet`) and reused across pets, then on the ZK path the in-circuit consent pubkey `Poseidon(Ax,Ay)` is the SAME across all the person's pets → **re-links the per-pet addresses** through the shared consent key, defeating mitigation 2 for the ZK path.
- Worse: `ConsentKeyRegistry.keyOf` is keyed by `userWallet` (the per-pet address), so you'd need a separate bind per pet anyway — but if the same BabyJubjub key is bound to multiple per-pet wallets, `keyOf[walletA]==keyOf[walletB]==Poseidon(samekey)` publicly links walletA and walletB on-chain.
**Fix:** Make the BabyJubjub consent key **per-pet** (derive `m/.../{petIndex}` analog in the BabyJubjub domain), bind a distinct key per per-pet wallet, and state this in §1.10/§6.6/§13.7 mitigation 2. Otherwise the headline per-pet-unlinkability mitigation is silently defeated on the ZK path by a shared consent key.

### P-6 (Low) — DPIA "refresh" is asserted but no doc states WHO owns it / the controller allocation for the verification event
research/07 §4.1 flags controller/processor allocation as "genuinely unresolved"; research/12 §5.3(4) says "verifier as (joint) controller for the attestation it records." But none of the canonical docs (arch §11.1/§13.7) pin the controller relationship for the verification event (is the verifier a controller, the platform a joint controller?). This matters for the erasure-propagation duty (P-2). **Fix:** add one sentence to §13.7: "the verifier is controller for the attestation it records; the platform is joint controller for the relay + receipt; cross-controller erasure propagation (§11.6) covers verifier `verification_records`/consent copies."

### Privacy verdict (overall)
**The on-chain verification linkage is correctly identified as pseudonymous personal data and is acknowledged in §11.1 + §13.7 + the (refreshed) DPIA scope, with a coherent mitigation ladder (ZK-default, fresh-per-pet subject, ZK-v2 nullifier-instead-of-subject, off-chain deletable receipts, permissioned-chain preference).** The mitigations are *individually* sound and consistently named. **However the verdict is QUALIFIED-NEGATIVE for ship-as-is** on three concrete defects: (P-3) the "ZK-for-Art.9" rule is a default, not enforced, and the normal-path `purpose==recordType` re-introduces a dictionary-reversible Art. 9 marker on-chain — regulator-fatal; (P-5) the BabyJubjub consent key is not per-pet, silently defeating the fresh-per-pet-address mitigation on the ZK path; (P-2) the on-chain `ConsentKeyRegistry` linkage is unenumerated in DPIA scope and verifier-side erasure propagation has no concrete `ownerId→verifier` index. Fix P-3/P-5/P-2 and the privacy posture is defensible (still "mitigation, not safe harbour," per research/07 — correctly stated everywhere).

### B-3 (High) — Under-specified blockers for coding (the brief's "who relays / how verifier learns BabyJubjub key / how purpose chosen")

But `businesses` registry is keyed by `businessId`, and the consent carries `relayer` (a signer ADDRESS), not a businessId. The lookup `relayer-address → businessId → apiBaseUrl` requires `businesses.signerAddresses[]` to include the relayer address — it does (§9.1), but the reverse index isn't specified and a relayer could be whitelisted (`VERIFY:`) without being in any `businesses.signerAddresses[]`. **Fix:** state that the relayer wallet MUST be registered in `businesses.signerAddresses[]` (or a dedicated `verifierAddresses[]`) for the relay to resolve, and reject if unresolvable.
- **How the verifier/circuit learns the user's BabyJubjub key:** the prover (`dogtag-prover-rs`) needs `babyJubPubKey{Ax,Ay}` in the witness (impl §3.10 line 625). The user signs with EdDSA and sends `sig` (R8x,R8y,S) via the relay — but the **pubkey (Ax,Ay) is not in the `VerificationConsent` struct and not obviously in the relayed payload.** §11.8(g) ZK branch calls `prover.prove({...eddsaSig:sig})` without passing the pubkey. The circuit needs Ax,Ay as private inputs AND the on-chain `keyOf[subject]==Poseidon(Ax,Ay)` check. **Fix:** specify that the mobile sends `{babyJubPubKey, eddsaSig}` in the consent submission (the pubkey is public anyway, bound in `ConsentKeyRegistry`), and the prover reads it; or the verifier reads `keyOf[subject]` on-chain (but that's a hash, not Ax,Ay — can't reconstruct). - **How `purpose` is chosen/validated:** see C-2/C-3. ### B-4 (Medium) — Prover input `leafValues/leafSalts/merklePath` provenance unspecified for ZK-with-no-import
In ZK mode "no raw data on chain" AND the verifier receives no raw doc (research/12 §1.3). But the prover needs the **private leaf values, salts, typeTags, keyPath-hashes, and the Poseidon Merkle path** (impl §3.10 line 625, §11.8(d)). If the verifier holds no doc, **where does the witness come from?** The witness must be built ON THE DEVICE (the only party with the cleartext doc + salts), not on the verifier backend. But impl §3.9/§11.8(g) build the proof on the **verifier backend** (`prover.prove(...)`), which contradicts "ZK = verifier receives no raw data." **This is a core architectural contradiction:** either (a) the device sends the private witness inputs to the verifier (then the verifier DOES see the cleartext leaves+salts — not "no raw data"), or (b) the device runs the prover and sends only `(a,b,c,pub)` to the verifier to submit. The docs say the prover runs server-side (`crates/dogtag-prover-rs` in the business backend, §3.10) — which **breaks the ZK privacy claim.** **Fix:** Decide and document: for true "verifier sees no data," **the proof must be generated on the mobile device** (ark-circom/rapidsnark mobile, or the WASM prover), and the verifier only relays/submits `(a,b,c,pub)`. If proving stays server-side, ZK mode does NOT achieve data-minimization vs the verifier (only vs the chain) — and §13.7 mitigation 1 / research/12 §1.3's "groomer never holds the cert" claim is FALSE. This must be resolved before Phase 2.5/3 coding. (Borderline ZK-auditor overlap, but it's a systems/privacy architecture decision, so flagged here.)
